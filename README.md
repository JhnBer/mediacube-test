# Инструкции по запуску

## Инфа по запуску

Скопировать .env.example -> .env

```shell
cp .env.example .env
```

Запустить проект
```shell
docker compose up -d --build
```

Зависимости
```shell
docker exec -it app composer i
```

Миграции
```shell
docker exec -it app php artisan migrate
```

Сид бд (около 3х минут)
```shell
docker exec -it app php artisan db:seed
```

Тесты
```shell
docker exec -it app php artisan test
```

## Структура

- Для ролей [UserRole](app/Enums/UserRole.php) enum + policy, так как роли и права базовые. 
Для сложных, скорее всего, взял бы spatie laravel-permission.

## Обоснование архитектурных решений

### Контроллеры и сервис-классы

Для постов и статистики логика была снесена в сервис-классы, чтобы оставить контроллеры тонкими.
В Comment контроллере показалось, что это будет лишним, он и так достаточно тонкий.

### FormRequest и Data Transfer Objects

В контроллерах используются FormRequest. В целом для чистоты почти всегда классы, 
даже когда достаточно было бы стандартного Request. 

Для демо добавил spatie data для жирных методов, по типу листинга и поиска.
Чтобы избавиться от типичного $options или длинного списка параметров.
Для идентичности можно было бы привести весь сервис класс к подобному виду и 
работе через DTO. Для тестового показалось избыточным, да и на деле можно вводить по
мере надобности.

Для отображения используются PostResource / CommentResource / UserResource.

### Кеширование, индексы, оптимизация

Для кеширования добавил использование Cache фасада и теги.
Для аналитики — сброс можно сделать ручками. Для поиска и списка сбрасывается через ttl.
В сервисе есть флаг для работы без кеша, для удобства локальных тестов.
Во избежание проблем сериализации приводится от ресурса через ->response()->getData() к чистым массивам.

Индексы находятся в отдельной папке в миграциях. Есть возможность прогнать или откатить
только индексы в любое время. Для разработки в сидере вызывается удаление индексов,
а затем их повторное создание после выполнения db:seed, для экономии времени.
Актуально больше для первого заполнения бд, если данных очень много.


### Защита от одновременного создания двух постов с одним тайлом

Есть реализация подобного поведения в тесте.
Если два запроса прошли валидацию на этапе FormRequest запроса к бд, а затем один из них выполнился до конца
и успешно создал пост, то для второго отлавливаем ошибку из бд и возвращаем сообщение об ошибке.
Можно сделать через Cache::lock, но показалось избыточным.

### Справочники meta/roles

Не особо понял, что именно там требовалось. Можно добавить какие-то label и красиво
отдавать во фронт, но пока просто отдаётся :)

Если бы он был супер-мега-большим и там билдились какие-то описания для ролей или что-либо
еще из конфигов, то возможно можно было бы накинуть дополнительный кеш на какой-то финальный
метод, который готовит данные для фронта.
